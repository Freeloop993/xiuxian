# 龙虾数字分身世界 - MVP 需求与执行文档（OpenClaw 兼容）

## 1. 背景
项目目标是让用户的 AI 数字分身（龙虾）不只进行“工作/生活”对话，而是在一个可持续运行的游戏世界中进行角色扮演（Role Play），并具备日常成长、社交和娱乐体验。

MVP 阶段优先做“可玩闭环”：
- 用户每天能通过聊天驱动龙虾行动。
- 龙虾在世界中形成连续状态变化（地点、关系、心情、资源）。
- 用户能感受到“这是我的分身”，而非一次性对话机器人。

## 2. 产品需求（Requirements）

### 2.1 核心玩法需求
1. 玩家拥有唯一龙虾分身。
2. 每个游戏日分为 4 个时段：早、午、晚、夜。
3. 每个时段执行 1 个动作：工作、社交、休息、娱乐、冒险。
4. 动作触发属性变化和事件文本。
5. 龙虾有基础人格，影响动作偏好与事件结果。

### 2.2 角色与状态需求
1. 龙虾基础属性：`体力`、`心情`、`金币`、`社交`、`探索欲`。
2. 初始人格（MVP）：`稳健型`、`冒险型`、`社交型`。
3. 状态持久化：跨会话保存，支持次日继续。

### 2.3 世界需求
1. MVP 世界为像素风小镇（表现层可先简化）。
2. 首批地点：`家`、`工坊`、`酒馆`、`海边`、`竞技场`。
3. 每个地点维护独立事件池。

### 2.4 内容需求
1. 事件条目：MVP 至少 30 条（目标 50 条）。
2. 连续剧情链：至少 10 条。
3. 结局倾向：`卷王`、`享乐`、`平衡`（按累计行为判定）。

### 2.5 对话交互需求
1. 用户可输入自然语言，例如“今晚去酒馆”或“你自己决定”。
2. 系统支持意图解析：文本 -> 标准动作。
3. 系统返回：行动结果 + 属性变化 + 下个时段建议。

## 3. MVP 目标（Goals）

### 3.1 业务目标
1. 新用户 1 分钟内完成创建并进入第一天。
2. 单日留存验证：用户愿意连续体验 3 天以上。
3. 形成“分身差异感”：不同人格出现可感知行为差异。

### 3.2 体验目标
1. 每次交互耗时控制在 10-30 秒响应。
2. 对话与世界状态强绑定，避免“只聊天不改变世界”。
3. 提供轻量叙事反馈，确保每天有新鲜感。

### 3.3 技术目标
1. 后端实现稳定状态机，确保可回放和可追踪。
2. 与 OpenClaw 完整打通单轮会话回调闭环。
3. API 与事件格式可扩展到多人和复杂玩法。

## 4. OpenClaw 兼容要求（必须满足）

### 4.1 接入边界
1. OpenClaw 作为消息入口与多渠道网关。
2. 游戏服务作为“世界状态引擎”，负责规则计算与持久化。
3. 职责分离：
   - OpenClaw：收发消息、路由、会话上下文传递。
   - Game Server：意图解析、状态推进、事件选择、文案返回。

### 4.2 会话与用户映射
1. 使用 `channel_user_id` + `channel` 映射内部 `user_id`。
2. 每条输入必须可关联 `session_id`（如渠道会话 ID）。
3. 多端同一用户统一到同一龙虾分身。

### 4.3 消息协议（MVP）
#### 输入（来自 OpenClaw）
```json
{
  "session_id": "string",
  "channel": "discord|telegram|whatsapp|...",
  "channel_user_id": "string",
  "text": "今晚去酒馆",
  "timestamp": "ISO8601"
}
```

#### 输出（返回 OpenClaw）
```json
{
  "reply_text": "你和老友在酒馆聊到深夜，心情+2，社交+1，体力-1。",
  "game_state": {
    "day": 3,
    "timeslot": "night",
    "location": "酒馆",
    "stats": {
      "energy": 6,
      "mood": 8,
      "gold": 12,
      "social": 5,
      "explore": 4
    }
  },
  "next_suggestions": ["回家休息", "去海边散步", "明早去工坊打工"]
}
```

### 4.4 失败与降级策略
1. 意图不明确时返回澄清问题，不推进时段。
2. 事件服务异常时，返回兜底文案并记录错误日志。
3. OpenClaw 回调重试需保证幂等：同一 `session_id + timestamp` 不重复结算。

### 4.5 审计与可观测性
1. 记录每轮：输入文本、解析动作、事件 ID、属性变化。
2. 记录错误类型：解析失败、状态冲突、数据库异常。
3. 保留最小可回放数据，支持复盘用户路径。

## 5. 范围边界（MVP 不做）
1. 实时多人同屏战斗。
2. 复杂交易市场和玩家对玩家经济系统。
3. 重美术资产生产（先文本 + 简化像素场景）。
4. 长上下文向量记忆系统（先结构化短中期记忆）。

## 6. 执行任务（Tasks）

### 6.1 第一阶段：规则与数据（D1-D2）
1. 定义属性、地点、动作、时段、人格枚举。
2. 产出事件 JSON 规范。
3. 编写首批 20-30 条事件数据。

交付物：
- `docs/game-rules-v0.1.md`
- `data/events.seed.json`

### 6.2 第二阶段：后端闭环（D3-D5）
1. 完成状态推进引擎（输入 -> 动作 -> 事件 -> 状态）。
2. 实现基础数据库 schema 与迁移。
3. 提供 REST API：创建角色、推进时段、查询状态。

交付物：
- `docs/api-spec-v0.1.md`
- `db/schema.sql`
- `src/server/*`

### 6.3 第三阶段：OpenClaw 接入（D6-D7）
1. 增加 webhook 入口，接收 OpenClaw 消息。
2. 完成用户映射（channel_user_id -> user_id）。
3. 返回统一响应格式并完成幂等处理。

交付物：
- `src/integrations/openclaw.ts`
- `docs/openclaw-contract.md`

### 6.4 第四阶段：前端与可视化（D8-D10）
1. 完成像素风小镇最小地图展示。
2. 展示龙虾状态条、当前地点、今日轨迹。
3. 提供最近事件日志面板。

交付物：
- `src/web/*`

### 6.5 第五阶段：内容扩充与调优（D11-D14）
1. 事件扩展到 50 条以上。
2. 增加 10 条剧情链触发条件。
3. 完成基础平衡测试与稳定性测试。

交付物：
- `docs/balance-notes-v0.1.md`
- `docs/test-report-mvp.md`

## 7. MVP 验收标准
1. 用户完成创建后可连续推进 7 个游戏日，状态无异常中断。
2. 任意一条自然语言输入可在 1 次内解析或给出澄清。
3. OpenClaw 渠道消息往返成功率 >= 99%（测试环境）。
4. 至少 3 种人格在 3 日行为轨迹上体现统计差异。

## 8. 开发顺序（本周执行建议）
1. 先完成规则文档 + schema + 事件 seed。
2. 再做状态机与 API。
3. 最后接 OpenClaw webhook，完成端到端联调。
